# REFLECTION.md — Бие даалт 14

## 1. Аль assertion хамгийн үнэ цэнэтэй санагдсан вэ? Яагаад?

Хамгийн үнэ цэнэтэй assertion нь **email format-ийн бизнес дүрмийн шалгалт** буюу `pm.expect(d.email).to.match(/@/)` гэсэн тест байсан. Status code шалгах нь хангалттай биш — API 200 буцаасан ч буруу өгөгдөл буцааж болдог. Email field байгаа эсэхийг шалгахаас гадна агуулга нь зөв форматтай эсэхийг шалгах нь API-ийн "contract"-ийг бодитоор баталгаажуулдаг. Жишээлбэл, хэрэв backend-д email validation эвдэрвэл `@` тэмдэггүй утга буцаах боломжтой — энэ тестгүйгээр тийм алдааг олоход хэцүү байх байсан. Ерөнхийдөө status + schema + business rule гэсэн гурван давхаргын хамт ашиглавал API-ийн найдвартай байдлыг бүрэн шалгаж чадна гэдгийг ойлгосон.

## 2. Negative test-ийн жишээгээ дэлгэрэнгүй тайлбарла — яг ямар алдааг олох вэ?

Negative test-ийн хувьд `GET /users/999999` endpoint-д байхгүй хэрэглэгчийн ID илгээж **404 Not Found** буцаж ирэхийг шалгасан. Энэ тест нь дараах алдааг илрүүлэх чадвартай:

- Хэрэв API 404 оронд 200 буцааж, хоосон object `{}` өгвөл — backend-д "not found" handler дутуу гэсэн үг
- Хэрэв 500 Internal Server Error буцааж ирвэл — ID-г validate хийхгүйгээр database query шууд явуулж байгаа гэсэн үг
- Response body-д `message` field байгаа эсэхийг шалгаснаар клиент тал хэрэглэгчид ямар алдаа гарсныг харуулж чадах эсэхийг баталгаажуулдаг

"Хэвийн ажиллах" тестүүд дангаараа хангалтгүй — алдааны тохиолдолд API хэрхэн хариу өгдөг нь мөн адил чухал contract болдог.

## 3. Postman дотор амжилттай ажиллаж байсан тест Newman-д орвол fail болсон уу? Яагаад?

Тийм, Newman-д анх ажиллуулахад **бүх 8 request** `Invalid URI "http:///..."` алдаагаар fail болсон. Шалтгаан нь `env.dev.json` файлд `baseUrl`-ийн value хоосон `""` байсан явдал юм. Postman дотор environment variable-г UI-аас шууд тохируулдаг тул ажиллаж байсан, гэвч export хийхэд runtime утга нь хадгалагддаггүй — зөвхөн анхны (хоосон) утга export болдог байжээ. Энэ нь надад чухал сургамж болсон: Newman-д ажиллуулахын өмнө `env.json` файлын утгуудыг заавал гараар шалгах хэрэгтэй. `baseUrl`-г `"https://dummyjson.com"` гэж тохируулсны дараа бүх тест амжилттай болсон.

## 4. Token эсвэл secret-ыг хэрхэн зохицуулсан вэ?

Token-г environment variable-аар дамжуулах аргыг ашигласан. Login request-ийн Test script-д `pm.environment.set("token", json.accessToken)` гэж бичиж, response-оос авсан token-г environment-д хадгалсан. Дараагийн request-үүд `{{token}}` хувьсагчаар Authorization header-т ашигласан.

`env.dev.json` болон `env.ci.json` файлуудад token-ийн утгыг хоосон `""` байлгасан — бодит token-г файлд хадгалаагүй. README.md-д "token автоматаар Login request-аас авагдана" гэж тайлбарласан. GitHub-д secret commit хийхгүй байх нь маш чухал — нэгэнт commit хийгдвэл history-д үлддэг тул устгахад хэцүү болдог.

## 5. API өөрчлөгдвөл collection-ийн аль хэсэг хамгийн их эвдрэх вэ? Яаж бууруулах вэ?

Хамгийн эмзэг хэсэг нь **response schema-г шалгах assertion-ууд** юм. Жишээлбэл, `accessToken` field-ийн нэр өөрчлөгдвөл (жишээ нь `jwt` болвол) Login-ий chain бүхэлдээ эвдэрч, token авч чадахгүй болсноор `Get My Profile` зэрэг бүх дараагийн request-үүд fail болно. Мөн endpoint-ийн URL бүтэц өөрчлөгдвөл (жишээ нь `/posts/add` → `/posts/create`) бүх тухайн request fail болно.

Энэ эрсдлийг бууруулах арга:
- **Collection-д version tag нэмэх** — аль API version-д тохирсон collection болохыг тэмдэглэх
- **Schema validation** ашиглах — property нэрийг хатуу шалгахын оронд Ajv зэрэг schema validator ашиглах
- **CI-г тогтмол ажиллуулах** — API өөрчлөгдөх бүрт автоматаар тест ажиллаж, эвдрэлийг эрт илрүүлэх
- **baseUrl-г environment-д хадгалах** — URL өөрчлөгдөхөд зөвхөн нэг газар засахад хангалттай болгох
